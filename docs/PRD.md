# 产品需求文档 (PRD)

## 1. 产品概述

### 1.1 产品名称

**ContentGuard — 智能数字媒体内容审核工作台**

### 1.2 一句话描述

面向游戏宣传素材团队，提供「YOLO 机器初筛 + 人工复核 + 结果留档」的一站式内容审核工作台。

### 1.3 目标用户

| 角色 | 描述 | 核心诉求 |
|---|---|---|
| 内容审核员 | 日常负责审核游戏宣传图片和视频的工作人员 | 快速定位问题帧、统一审核标准、追溯审核记录 |
| 项目负责人 | 管理多个素材项目的负责人 | 查看审核统计、导出审核报告、掌握整体合规情况 |

### 1.4 核心价值主张

> **让机器做初筛，让人做决策。** 不再逐帧看视频——YOLO 自动标记风险目标，审核员只需对机器存疑的内容做最终判断。每条审核结论都有证据帧支撑，可追溯、可复核。

## 2. 业务场景

### 场景 1：新宣传视频上线前审核

**背景：** 市场部制作了一段游戏角色宣传视频，需要在上线前确保画面中不出现违规露出。

**流程：**
1. 审核员打开 ContentGuard 工作台
2. 上传视频 `hero_trailer.mp4`，填写项目名称 "夏日活动宣传片审核"
3. 系统自动创建任务，返回任务编号
4. 后台 YOLO 对视频每秒采样一帧，检测目标
5. 系统按审核规则给出初筛结论：**待复核**（在第 15 秒检测到 person，置信度 0.52，处于灰色区间）
6. 审核员点开任务，查看第 15 秒的证据帧图片
7. 审核员判断画面中只是远景路人，无违规风险，人工修改结论为 **通过**
8. 系统将人工审核结果写回 JSON 报告，任务完成

### 场景 2：批量图片快速筛查

**背景：** 项目组收到一批游戏截图（50 张），需要逐张检查是否包含敏感内容。

**流程：**
1. 审核员逐张上传图片（或拖拽批量上传）
2. 每张图片创建一个独立任务
3. YOLO 自动检测每张图片
4. 审核员在任务列表中按状态筛选：「不通过」的优先复查，「待复核」的逐一确认，「通过」的批量验收
5. 全部审核完成后，导出 CSV 统计报告交付项目负责人

### 场景 3：历史任务回溯

**背景：** 一周前审核过的视频，需要重新打开确认某条审核结论是否合理。

**流程：**
1. 审核员在任务列表中找到历史任务（支持按项目名称搜索）
2. 点击任务，页面恢复当时的审核结果、证据帧和人工审核记录
3. 如果需要修改结论，审核员可以再次执行人工审核操作
4. 系统保留审核修改历史（版本记录），标注每次修改的时间、审核人和理由

## 3. 核心流程

```text
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 上传素材  │───▶│ 创建任务  │───▶│ 后台分析  │───▶│ 展示结果  │───▶│ 人工审核  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
                                                                        │
                                                                        ▼
                                                            ┌──────────────────┐
                                                            │ 输出报告 + 证据帧  │
                                                            └──────────────────┘
```

### 详细步骤

| 步骤 | 操作者 | 动作 | 系统响应 |
|---|---|---|---|
| 1. 上传 | 用户 | 选择图片/视频，填写项目名称 | 保存文件，生成 job_id |
| 2. 创建 | 系统 | 创建 `job.json`，状态 = `created` | 返回 job_id |
| 3. 排队 | 系统 | 任务进入后台队列，状态 = `queued` | 页面显示 "排队中" |
| 4. 分析 | 系统 | YOLO 检测 → 审核规则评估，状态 = `running` → `completed` | 生成 `analysis_report.json` + 证据帧 |
| 5. 展示 | 系统 | 前端展示检测结果 + 证据帧 + 初筛结论 | 页面渲染结果卡片 |
| 6. 复核 | 用户 | 保留或修改审核结论 | 写回 JSON，记录审核人 + 时间 |
| 7. 输出 | 用户 | 查看/导出审核报告 | 页面展示完整报告 |

## 5. 审核规则

### 5.1 默认配置

```json
{
  "risk_classes": ["person"],
  "reject_confidence": 0.60,
  "review_confidence": 0.35,
  "min_evidence_frames": 1
}
```

### 5.2 判定逻辑（优先级从高到低）

```text
规则 1 — 不通过 (reject)
  条件: 任意帧中检测到的目标满足:
    - 类别在 risk_classes 中
    - 置信度 >= reject_confidence (0.60)
  示例: "第 15 帧检测到 person，置信度 0.85 >= 0.60 → 不通过"

规则 2 — 待复核 (review)
  条件 A: 任意帧中检测到的目标满足:
    - 类别在 risk_classes 中
    - 置信度 >= review_confidence (0.35) 且 < reject_confidence (0.60)
  条件 B: 同一类别在不同采样帧的置信度波动 > 0.4
  示例: "第 8 帧 person 置信度 0.45，第 20 帧 person 置信度 0.52 → 待复核"

规则 3 — 通过 (pass)
  条件: 不满足规则 1 和规则 2，且处理过程正常
  示例: "未检测到风险类别目标，审核通过"
```

### 5.3 特殊场景处理

| 场景 | 处理方式 |
|---|---|
| 所有帧都未检测到任何目标 | verdict = pass，理由："未检测到任何目标" |
| 模型加载失败 | status = failed，error = "YOLO 模型加载失败：模型文件不存在或损坏" |
| 视频无法读取 | status = failed，error = "无法读取视频文件：文件损坏或格式不支持" |
| 采样帧数为 0 | status = failed，error = "视频采样失败：视频时长可能为 0" |

## 6. 验收标准

| 编号 | 验收项 | 验证方式 |
|---|---|---|
| AC01 | README 可成功启动项目，`/api/health` 返回 `{"status":"ok","model_ready":true}` | 按 README 操作，浏览器访问 health 接口 |
| AC02 | 上传合法 jpg 图片 → 返回 job_id → status 从 created 走到 completed | 上传图片，观察状态变化 |
| AC03 | 上传合法 mp4 视频 → YOLO 采样检测 → 生成证据帧图片 | 上传视频，查看 keyframes 目录 |
| AC04 | 包含高置信度 person 的图片，审核结果为 "不通过" | 用包含人物的图片测试 |
| AC05 | 页面展示原始素材（图片/视频）、检测结果表格、证据帧缩略图 | 截图验收 |
| AC06 | 人工修改审核结论后，刷新页面结论保持一致（已持久化） | 修改 → 刷新 → 验证 |
| AC07 | 上传空文件 → 返回 400 + `{"ok":false,"error":"..."}` | curl 测试 |
| AC08 | 上传 .exe 文件 → 返回 400，提示格式不支持 | curl 测试 |
| AC09 | 删除 completed 任务 → 成功，outputs 目录被清理 | 删除后检查目录 |
| AC10 | 删除 running 任务 → 返回 409，提示无法删除 | curl 测试 |
| AC11 | 历史任务可重新打开，展示完整结果和审核结论 | 上传 → 完成 → 关闭 → 重新点击任务 |
| AC12 | 失败任务在页面显示红色 "失败" 标签 + 具体错误信息 | 故意让模型缺失，上传文件触发失败 |
| AC13 | 正常测试 ≥ 5 条，异常测试 ≥ 5 条，Bug 记录 ≥ 2 条 | 检查 TEST_REPORT.md 和 BUG_RECORD.md |

## 7. 非功能需求

### 7.1 性能

| 指标 | 目标 |
|---|---|
| 单张图片检测 | < 5 秒 |
| 10 秒视频采样检测 | < 30 秒 |
| 页面首次加载 | < 2 秒 |
| 任务状态轮询间隔 | 2 秒 |

### 7.2 可靠性

- 任何异常不得导致 Flask 进程崩溃
- 任务失败必须落盘 `job.json`（含 error 字段），不能只在控制台输出
- 删除操作禁止越权删除非 outputs 目录下的文件（路径遍历防护）

### 7.3 可用性

- 所有错误信息使用中文，包含具体原因和排查建议
- 页面在以下状态均有对应 UI：加载中、空列表、成功、失败、网络断开
- 上传大文件时页面不卡死（后台异步处理）

### 7.4 兼容性

- 浏览器：Chrome / Edge（最新两个大版本）
- 分辨率：1366×768 及以上

## 8. 风险与约束

### 8.1 已知风险

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| YOLO 模型检测速度慢 | 视频分析耗时长 | 采样间隔默认 1 秒，必要时调大到 2 秒 |
| 大视频文件上传超时 | 上传失败 | 限制 500MB，前端显示进度 |
| 团队首次使用 Flask + YOLO | 开发进度慢 | 优先完成最小闭环，P1/P2 降级 |
| FFmpeg 未安装或版本不兼容 | 视频读取失败 | 方向 A 不强依赖 FFmpeg，OpenCV 兜底 |

### 8.2 技术约束

- 必须使用 Flask（不可换 Django/FastAPI）
- 必须使用 Ultralytics YOLO（不可换其他检测框架）
- 前端不可使用 React/Vue 等框架（Vanilla JS only）
- 数据存储仅限本地文件系统 + JSON（不可引入数据库）

### 8.3 范围外（本期不做）

- 用户登录和权限管理
- 视频实时流处理
- 自定义模型训练（使用预训练 yolo11n.pt）
- 邮件或消息通知
- 移动端适配
- Docker 容器化部署

## 9. 附录：任务状态定义

| 状态 | 含义 | 触发条件 |
|---|---|---|
| `created` | 任务已创建，等待处理 | POST /api/jobs 成功后 |
| `queued` | 任务已进入处理队列 | 后台线程即将开始处理 |
| `running` | 正在执行 YOLO 检测 | 后台线程开始调用 Detector |
| `completed` | 检测和审核已完成 | 审核引擎输出结论，JSON 已写入 |
| `failed` | 处理过程中出现异常 | 任何未捕获异常导致处理中断 |

状态流转：

```text
created → queued → running → completed
                            └→ failed (可重新 queued)
```
