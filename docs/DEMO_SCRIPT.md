# 演示脚本 — ContentGuard 项目验收演示

> **时长：8 分钟**
> **演示人：罗欢**
> **方式：浏览器实时操作 + 截图备用**

---

| 段落 | 时间 | 内容 | 时长 |
|---|---|---|---|
| 1 | 0:00-0:30 | 开场：一句话介绍项目 | 30s |
| 2 | 0:30-1:30 | 正常流程：上传 → 分析 → 结果 | 60s |

## 详细演示步骤

### 第 1 段：开场（0:00 - 0:30）

**操作：** 展示工作台首页

```
"大家好，我们组做的是 ContentGuard —— 智能数字媒体内容审核工作台。

面向游戏宣传素材团队，解决三个痛点：
第一，长视频逐帧审核效率低——我们用 YOLO 自动检测；
第二，审核标准不统一——我们配置规则引擎统一判定；
第三，审核结果难追溯——我们保存证据帧和完整报告。

一句话：机器初筛，人做决策，全程留档。"
```

**展示要点：**
- 页面整体布局清晰（左侧任务列表 + 右侧内容区）
- 空状态友好提示

---

### 第 2 段：正常流程（0:30 - 1:30）

**操作：** 上传一张包含人物的测试图片

```
"我们来看完整流程。"
```

1. 点击上传区域 / 拖拽图片 → 输入项目名称 "演示测试"
2. 点击上传 → 左侧任务列表出现新任务，状态 "排队中"（黄色）
3. 1~2 秒后状态变为 "处理中"（蓝色）
4. 3~5 秒后状态变为 "已完成"（绿色）
5. 点击任务 → 右侧展示完整结果

```
"从上传到审核完成，图片 5 秒内，视频按采样帧数线性处理。
这里可以看到，状态从 created → queued → running → completed，全程自动流转。"
```

**如果速度慢的备选方案：** 提前准备好已完成的任务，直接点开展示。

---

### 第 3 段：深度展示（1:30 - 2:30）

**操作：** 逐区域介绍结果页面

```
"现在看结果页面的四个区域："
```

**区域 1 — 原始素材：**
- 展示上传的图片或视频缩略图
- 显示文件名、类型、上传时间

**区域 2 — 检测结果表格：**
- 帧序号、时间戳、检测类别、置信度、边界框坐标
- 高亮高风险检测行（红色标注）

```
"这是 YOLO 在第 3 帧检测到的 person，置信度 0.85。
边界框精确标注了目标位置。"
```

**区域 3 — 证据帧：**
- 网格展示 1~3 张关键帧缩略图
- 点击可放大查看

```
"证据帧是置信度最高的检测画面，作为审核证据永久保存。
即使原视频删了，这张图也能说明当时为什么做出这个审核结论。"
```

**区域 4 — 审核结论卡片：**
- 自动结论：不通过（红色卡片）
- 结论理由：发现 person，置信度 0.85 >= 0.60

```
"系统自动判定为'不通过'，并给出了明确的理由。
现在看人工审核——如果审核员认为这个结论需要调整。"
```

---

### 第 4 段：人工审核（2:30 - 3:30）

**操作：** 修改审核结论

```
"如果审核员判断画面中的人物实际不构成风险——比如只是远景路人，
他可以直接在前端修改审核结论。"
```

1. 选择审核结论：点击 "通过" 按钮
2. 填写审核人：输入姓名
3. 填写备注：输入 "远景路人，无违规风险"
4. 点击提交 → Toast 提示 "审核结果已保存"
5. **关键操作：** 刷新页面 → 重新打开同一个任务
6. 审核结论显示为 "通过"，manual_review 字段已更新

```
"审核结果已经持久化到 JSON 文件。即使关闭浏览器、重启服务，
结论都不会丢失。任何历史任务都可以随时重新打开。"
```

---

### 第 5 段：异常处理（3:30 - 4:30）

**操作：** 演示三种异常场景

#### 场景 A：空文件上传

```bash
# 用 curl 演示，或在页面上传空文件
curl -X POST http://127.0.0.1:7880/api/jobs \
  -F "file=@empty_file.txt" \
  -F "project_name=test"
```

```
"上传空文件，返回 400 + 中文错误信息：'上传文件为空，请选择有效文件'。
页面同步显示红色错误提示条，不会白屏。"
```

#### 场景 B：不支持格式

```
"上传 .exe 文件：返回 400，提示'不支持的文件格式 .exe，支持格式：jpg, png, mp4...'"
```

#### 场景 C：模型缺失

```
"如果模型文件意外丢失——
YOLO 不会让整个服务崩溃。任务状态变为 'failed'，
页面显示红色标签和具体错误信息：'YOLO 模型加载失败'。
job.json 仍然保留，error 字段有完整堆栈，方便排查。"
```

**要点：** 强调三种异常都是「中文提示 + 不崩溃 + 有记录」。

---

### 第 6 段：历史任务与任务管理（4:30 - 5:30）

**操作：** 展示任务列表和任务管理功能

1. 任务列表展示多个历史任务（至少 5 个），按时间倒序
2. 展示颜色标签区分状态（绿=完成，红=失败，黄=排队，蓝=处理中，灰=创建）
3. 如果有搜索功能：输入项目名称关键词 → 实时筛选
4. 删除一个 completed 任务 → 确认弹窗 → 删除成功 → 列表刷新
5. 尝试删除 running 任务 → 返回 409 "任务正在处理中，无法删除"

```
"任务列表支持按时间排序和项目名称搜索。
删除操作有安全保护：正在处理的任务不允许删除，防止误操作。"
```

---

### 第 7 段：工程交付（5:30 - 6:30）

**操作：** 展示项目文档体系（切换 VS Code 或文件管理器）

"测试覆盖了正常和异常两类场景，
每个 Bug 都记录了现象、复现步骤、根因和修复方案。

## 备用方案

### 如果遇到以下情况

| 问题 | 应对 |
|---|---|
| **Flask 启动失败** | 提前 10 分钟启动，确认 `/api/health` 正常 |
| **YOLO 推理太慢** | 提前准备好已完成的任务，演示时直接打开结果 |
| **网络断开** | 使用 `127.0.0.1` 本地访问，不依赖外网 |
| **浏览器卡死** | 准备页面截图备用（`screenshots/` 目录） |
| **操作失误** | 提前演练 2 遍以上，熟练每个点击路径 |

### 提前准备清单

- [ ] Flask 服务已启动，health 接口正常
- [ ] 至少 3 个已完成的任务（不同审核结论：pass、review、reject）
- [ ] 至少 1 个 failed 任务（用于展示异常处理）
- [ ] `screenshots/` 目录下有各状态截图（备用）
- [ ] 浏览器已打开，页面已加载，无缓存问题
- [ ] VS Code 或文件管理器已打开 docs/ 目录
- [ ] 终端已打开，可用于 curl 演示异常场景
- [ ] 演示稿（本文档）已打印或第二屏幕打开
